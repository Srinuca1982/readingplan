# Deployment Guide — planner.zerolagautomation.com

Single-file study planner deployed via **GitHub Pages**, fronted by **Cloudflare Access** for restricted login.

## Architecture

```
Visitor → planner.zerolagautomation.com
        → Cloudflare DNS (proxied)
        → Cloudflare Access (email-OTP login gate)
        → GitHub Pages (serves dist/index.html)
```

The repo holds Python source; a GitHub Actions workflow runs `build.py` on every push to `main` and publishes `dist/` to GitHub Pages.

## One-time setup (you do these)

### Step 1 — Push the repo to GitHub

From this folder:

```bash
cd "/Users/srnivasu/Library/CloudStorage/OneDrive-Personal/Claude/readingplan"
git init
git add .
git commit -m "Initial commit — study planner with deploy workflow"
git branch -M main
```

Create a new repo on GitHub (any name; e.g. `readingplan`). Then push:

```bash
git remote add origin https://github.com/<your-username>/readingplan.git
git push -u origin main
```

> The repo can be **private** — GitHub Pages still works on private repos (with a Pro/Team plan) or you can make it public. Public is fine because Cloudflare Access protects the URL; the source code on GitHub is not the same as the hosted page.

### Step 2 — Enable GitHub Pages

On the GitHub repo page:

1. `Settings` → `Pages`
2. Under **Source**, select **GitHub Actions**.
3. The workflow at `.github/workflows/deploy.yml` will run automatically on the next push.

Wait ~1 minute. The Actions tab will show a successful run with two jobs: `build` and `deploy`. The deploy job's URL output gives you a temporary `<username>.github.io/<repo>` link — confirm it loads.

### Step 3 — Set up Cloudflare (DNS + Access)

If `zerolagautomation.com` is **not on Cloudflare yet**:

1. Create a free Cloudflare account at https://dash.cloudflare.com/sign-up
2. Add the site `zerolagautomation.com`.
3. Cloudflare will give you 2 nameserver addresses. Update them at your domain registrar (where you originally bought the domain). Wait for propagation (usually <1 hour).

If `zerolagautomation.com` is already on Cloudflare, skip ahead.

### Step 4 — Add the DNS record

In Cloudflare → DNS → Records → **Add record**:

| Field | Value |
|---|---|
| Type | `CNAME` |
| Name | `planner` |
| Target | `<your-github-username>.github.io` |
| Proxy status | **Proxied** (orange cloud — required for Cloudflare Access) |
| TTL | Auto |

This makes `planner.zerolagautomation.com` resolve via Cloudflare.

### Step 5 — Tell GitHub about the custom domain

On the GitHub repo:

1. `Settings` → `Pages` → **Custom domain** → enter `planner.zerolagautomation.com` → **Save**.
2. GitHub will validate (may take 5–10 minutes).
3. Tick **Enforce HTTPS** once it's available.

The CNAME file in `dist/` is auto-generated by `build.py`, so this stays consistent across rebuilds.

### Step 6 — Set up Cloudflare Access

In Cloudflare dashboard → **Zero Trust** (top-left dropdown) → **Access** → **Applications** → **Add an application** → **Self-hosted**.

| Field | Value |
|---|---|
| Application name | Study Planner |
| Session duration | 30 days (or your preference) |
| Application domain | `planner.zerolagautomation.com` |

**Identity provider:** by default, Cloudflare Access offers **One-time PIN** (email OTP) out of the box. You can also add Google / GitHub / Microsoft as identity providers later.

**Policies:** add a single Allow policy:

- Name: `Allow listed emails`
- Action: Allow
- Rule: Include → Emails → enter your email (e.g. `you@zerolagautomation.com`). Add additional teammates as needed.

Save. Done.

### Step 7 — Verify

Visit `https://planner.zerolagautomation.com` in an incognito window. You should see:

1. Cloudflare Access login screen → enter your email → receive 6-digit code via email → enter code.
2. Planner loads.

Total setup time: ~20 minutes (excluding nameserver propagation if needed).

## Day-to-day: how updates flow

Edit any `content_*.py` or `templates.py` locally → commit → push to `main`:

```bash
git add .
git commit -m "<what changed>"
git push
```

GitHub Actions rebuilds and redeploys in ~30 seconds. The planner at `planner.zerolagautomation.com` reflects the changes on next reload.

Your local browser run (`python3 planner.py`) continues to work unchanged — it still writes to OneDrive for offline study. The two outputs are identical.

## Troubleshooting

- **Pages deploy fails with permission error**: Settings → Actions → General → "Workflow permissions" → set to "Read and write permissions". Re-run.
- **DNS works but page shows GitHub 404**: confirm the `CNAME` file in `dist/` matches the custom domain set in GitHub Pages settings.
- **Cloudflare Access blocks even after correct OTP**: check the Allow policy's email rule actually contains your address (exact case-insensitive match).
- **Workflow uses an older planner output**: bump the workflow with a small change in `templates.py` and push; cache is per-commit so it'll rebuild fresh.

## Future enhancements

- **Google SSO** instead of email OTP: Cloudflare Zero Trust → Settings → Authentication → Add new → Google.
- **Multiple environments** (e.g., a staging URL): duplicate the workflow with a different output domain.
- **Server-side state** (sync progress across devices): would require moving off GitHub Pages to a host with a backend (Firebase, Supabase). Out of scope for this static deployment.
